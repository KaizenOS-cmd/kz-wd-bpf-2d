01
How to Use This SOP
Read before editing any business process

The Business Process Framework governs every HR transaction in Workday — from Hire to Terminate, Change Job to Compensation Change. A misconfigured BP can stall transactions for hundreds of employees, route approvals incorrectly, or bypass controls entirely. This SOP provides a repeatable, auditable approach to BP configuration and change management.

🔴 Critical
Business Process changes affect live transactions immediately upon saving — there is no "pending activation" step as with security. If you edit a BP while transactions are in-flight, those transactions continue on the old definition. New transactions use the new definition. Always coordinate BP changes with your HR Ops team to avoid mid-process confusion.
✅ Tip
Duplicate this document per business process. Name files using your convention (e.g., SOP_BP_Hire_Employee_v1.0_2026.html). For complex BPs with many variants, create one file per sub-process definition.

Core concepts before configuration

Workday business processes are structured sequences of steps that govern how HR transactions flow from initiation to completion. Every BP has a definition, one or more steps, and security policies controlling who can do what.

Condition Rules

Logic that controls whether a step is included, skipped, or routed differently based on worker or transaction data. The primary tool for building intelligent, conditional BPs.

BP Definitions vs. Templates

A BP Definition is a specific configuration attached to a BP type. A BP Template is a reusable structure. Most orgs configure Definitions directly — Templates are used for standardised multi-tenant deployments.

Effective Dating

BP changes are not effective-dated. A saved change applies to all new transactions immediately. Plan changes carefully around high-volume periods (month-end, open enrolment).

BP Inheritance

Child orgs can inherit BP definitions from parent orgs or override with their own. Understand your org's inheritance model before making changes — an edit at the wrong level can unexpectedly affect multiple entities.

04
Business Process Configuration — Step-by-Step
Editing and building BP definitions in Workday
1
Locate the Business Process Definition
Search for the BP by name (e.g., Hire Employee, Change Job). Navigate to Business Process task and select Edit Definition. Alternatively, use View Business Process Definitions to browse all BPs. Confirm you are editing the correct definition — many BPs have multiple definitions per org or country.
✅ Tip
Before editing, run View Business Process Definition and screenshot the current state. This is your rollback reference — Workday does not version BP definitions automatically.
2
Add or Edit a Step
In the BP definition canvas, click Add Step (or click an existing step to edit). Select the step type from the dropdown (see Section 02 reference table). Configure: Step name, security group assignment (who acts on this step), order/sequence, and whether the step is required or optional.
📋 Note
Step order matters but is not always sequential — Workday supports parallel steps (steps that run simultaneously). Mark steps as parallel where the business process design permits concurrent action, such as simultaneous IT provisioning and payroll notification.
3
Configure Approval Step Routing
For Approval steps, configure the routing target: Specific Security Group (e.g., HR Director group), Manager (direct or skip-level), or Approval Chain (sequential up the org hierarchy). Set the order (single approver vs. all must approve) and delegation rules.
⚠️ Warning
Approval Chain routing can create extremely long approval sequences for deep org hierarchies. Always set a maximum levels cap and test with your deepest org structure in sandbox before enabling in production.
4
Set Escalation & SLA Rules
On each Approval or To-Do step, configure Due Date Rules: number of days before the step is overdue, and escalation actions (re-assign, notify manager, alert HRIS). Navigate to Maintain Business Process Step Due Dates for org-level defaults. Override at the step level where specific SLAs apply.
5
Save & Validate Structure
Save the definition. Workday will surface validation warnings for common configuration errors (orphaned steps, missing security groups, invalid routing). Resolve all warnings before proceeding to condition rule configuration in Section 05. Do not test in production with open validation warnings.
🔴 Critical
Saving a BP definition applies it immediately to all new transactions of that type. If you are mid-configuration and not ready to go live, use the sandbox tenant only until the full configuration is complete and tested.
05
Condition Rules & Routing Logic
Making BPs intelligent — conditional steps, skip logic, and dynamic routing

Condition rules are the most powerful and most commonly misconfigured element of BPF. They determine whether a step is included, skipped, or routed differently based on worker or transaction attributes. Mastering condition rules is the difference between a rigid BP and an intelligent one.

1
Navigate to Condition Rule Configuration
Within the BP step, click Add Condition. Select Create a New Condition Rule or select an existing reusable rule. Name the rule descriptively — e.g., Condition: Salary Change > 10% of Band Midpoint. This name will appear in the BP definition and audit logs.
2
Define the Condition Logic
Condition rules use Workday's condition builder. Select the Data Source (e.g., Worker, Compensation Proposal, Position), the field (e.g., Proposed Salary, Employee Type, Country), the operator (equals, greater than, contains, etc.), and the value. Chain multiple conditions with AND/OR logic as needed.
✅ Tip
Build condition rules as standalone reusable objects — not embedded in a single BP. Reusable condition rules can be referenced across multiple BPs, reducing maintenance when business rules change. Navigate to Create Condition Rule as a standalone task.
3
Assign the Condition to the Step
Back in the BP definition, attach the condition rule to the step. Set the condition action: Skip Step if True or Include Step if True. Test your logic by tracing through the condition manually with test data before running a full end-to-end test. Document the expected outcome for each condition path.
4
Test All Condition Paths
For every condition rule, test both the TRUE and FALSE path in sandbox. Create test transactions that deliberately trigger each path. Confirm the BP routes correctly in all scenarios. Log each test result with the triggering data, expected path, and actual path.
⚠️ Warning
The most common BPF failure mode is an untested condition path. A condition rule that works for 95% of transactions can silently skip a mandatory approval step for a specific worker type or country. Test edge cases — international hires, rehires, contingent workers.

06
Notifications & To-Do Steps
Configuring communication and manual task steps
1
Configure Notification Steps
Add a Notification step to the BP. Select the target: Security Group, Role, or specific users. Choose the notification template — use Workday's delivered templates or create a custom notification via Create Notification. Configure whether the notification is sent immediately when the step is reached, or on BP completion.
💡 Rationale
Notifications are non-blocking — the BP continues regardless of whether the notification recipient acts. Use them for FYI communications (payroll, IT, facilities) where no action is required from the recipient. Use Approval steps where an action or sign-off is required.
2
Configure To-Do Steps
Add a To-Do step and assign to the relevant security group or role. Write the To-Do instruction text clearly — this is what the recipient sees in their Workday inbox. Set a due date. Configure whether the BP waits for To-Do completion (blocking) or continues regardless (non-blocking).
⚠️ Warning
Blocking To-Do steps will stall the entire BP until marked complete. Only use blocking To-Dos where the downstream step genuinely cannot proceed without the manual task being done (e.g., physical equipment provision before first day). For most cases, non-blocking To-Dos with SLA monitoring is the right pattern.
3
Configure Checklists for Parallel Tasks
For onboarding, offboarding, and similar multi-party workflows, use a Checklist step. Add multiple To-Do items targeting different groups (IT, Facilities, Payroll, HR) that run in parallel. Set individual due dates per item. The checklist step completes when all items are marked done — or configure it to continue after a defined subset complete.

08
Rollback & Risk Notes
Reversing BP changes and managing common failure modes
🔴 Critical
Workday does not maintain automatic version history for BP definitions. The only rollback option is to manually re-enter the previous configuration. This is why a pre-change screenshot (Section 04, Step 1) is non-negotiable. For complex BPs, export the full definition to a document before every change.
Rollback Procedure
  • Navigate to the BP definition and edit to restore the previous step configuration
  • Rebuild removed steps, re-add security group assignments, and restore condition rules from your pre-change documentation
  • Save and test with the same scenarios from Section 07
  • For in-flight transactions on the incorrect definition: assess each individually — some may need manual retraction and re-initiation on the corrected definition
  • Document the rollback event in the Change Log (Section 10)
Common Risk Scenarios

Mandatory approval step accidentally removedCriticalSecond reviewer must confirm all approval steps present before saving any BP change
